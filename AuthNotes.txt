 In password grant type
 you will provide username and password to the /token url and it will provide you the access token
 here there is no login paage given by the IDP provider
 you have to write your own login page
 or you need to pass username and password manually to the /token url

 This is a deprecated grant type in OIDC

                      CLIENT_CREDENTIAL GRANT TYPE

   IT is used when two services wants to connect with one another


   Absolutely! Client Credentials Grant is actually much simpler than Authorization Code Flow because there is no user involved at all.

   Think of it as application-to-application authentication.

   Real-world Example

   Suppose you have two microservices:

   Order Service
           │
           │ wants inventory information
           ▼
   Inventory Service

   There is no human using these services.

   The Inventory Service just needs to know:

   "Is this really the Order Service?"

   It doesn't care about Sahana or any other user.

   Authentication Flow
   Order Service
       │
       │ client_id
       │ client_secret
       ▼
   Keycloak
       │
   Validates client
       ▼
   Access Token
       │
       ▼
   Order Service
       │
   Authorization: Bearer token
       ▼
   Inventory Service

   Notice:

   ❌ No username

   ❌ No password

   ❌ No login page

   Token Request

   Call Keycloak:

   POST /realms/TodoRealm/protocol/openid-connect/token

   Body

   grant_type=client_credentials
   client_id=inventory-service
   client_secret=xxxxxxxx

   That's all.

   Keycloak returns

   {
     "access_token":"eyJ....",
     "expires_in":300,
     "token_type":"Bearer"
   }

   What is AntPathRequestMatcher?AntPathRequestMatcher is a core utility class in Spring Security used to match incoming HTTP requests against a specific URL pattern.
   It uses Apache Ant-style path patterns [1] (such as ? for a single character, * for zero or more characters, and ** for zero or more directories) to determine if a security rule applies to a request.
   Why do we need it here?When you call requestMatchers("/h2-console/**") as a raw string in Spring Boot 3, Spring Security tries to protect you by implicitly assigning a matcher type.
   It defaults to using MvcRequestMatcher, which relies on Spring MVC’s architecture to resolve paths.This causes the 401 Unauthorized loop with the H2 console for two critical reasons:1.
   The H2 Console is a Servlet, Not a Spring MVC ControllerThe H2 database console is not a standard Spring MVC endpoint.
   It is a standalone, legacy Java Servlet registered under your application.
   The Problem: MvcRequestMatcher only looks for paths registered within Spring MVC's dispatcher servlet handler mappings.
   The Result: Because Spring MVC doesn't "own" the /h2-console path, the raw string matcher skips it.
   The security engine falls back to your catch-all .anyRequest().authenticated() rule, which passes the request to your OAuth2 JWT validator.
   Since your browser isn't sending a JWT token to the console page, you get a 401 error.
